## Summary
This change implements the OAuth authentication flow (Google/GitHub) and basic UI routing as defined in the spec. The core logic for signing in, callbacks, and dashboard session reading is solid. However, a critical misnaming of the Next.js middleware file prevents the route protection from ever running, and there is currently no test coverage.

## Blockers
### 🔴 Misnamed middleware file, `job-pilot/proxy.ts:1`
**Problem**: The route protection logic is placed in a file named `proxy.ts`. Next.js convention requires this file to be named `middleware.ts` to be automatically detected and executed.
**Why it matters**: The middleware will not be picked up by Next.js, meaning all route protection (AC-5) will silently fail to execute. Unauthenticated users will be able to freely access protected routes like `/dashboard`, and authenticated users won't be redirected from `/login`.
**Suggested fix**: Rename `job-pilot/proxy.ts` to `job-pilot/middleware.ts`.

## Strengths

- Clean separation of concerns with server actions used for OAuth initialization.

## Test coverage
There are currently no tests for these features (TEST_SIGNAL is `none-yet`). A test runner should be configured to verify this critical authentication and proxy logic, especially since there's no safety net.
